# General Instructions

- after every change to the source code make sure the Agent.md file is updated
- after every new feature added, make sure there is a test for the feature (no tests for removing features)
- after changing the code, start a new dev server and ask the user to check the resulting app functionality


## Linting

Run `npm run lint` before committing JS changes. This parses all `web/**/*.js` files for syntax errors using acorn. The same check runs in CI before deploy.

Common issues it catches:
- `await` in non-async functions
- Mismatched brackets/parens
- Invalid ES module syntax

## Dependency Maintenance

- Keep npm dependencies current with `npm install <package>@latest` so `package.json` and `package-lock.json` stay in sync.
- Keep browser CDN dependencies pinned in `web/index.html`; check the upstream package version before changing those URLs.
- Google Analytics 4 is loaded from `web/index.html` with measurement ID `G-4Z9774J59Y`, and its `gtag('config', ...)` call is disabled when the browser reports Do Not Track as `1` or `yes`. Keep analytics limited to aggregate page usage and performance metrics; do not add custom analytics events or parameters carrying filenames, patient image metadata, voxel values, masks, screenshots, generated outputs, free text, or analysis results. Keep `scripts/test_index_html.mjs` and the privacy modal aligned if analytics changes.

## Architecture

- `web/js/lnm-app.js` — Main CALMaR app class. Owns the full chain: file load, brain extraction, lesion segmentation or uploaded/blank editable mask review, MNI registration, warp+resample bridge, selected-atlas overlap, FC network map, thresholding, subject-space atlas QC, version-label formatting, and the one-click `runFullPipeline()`
- `web/js/app/config.js` — Model config, version (bumped by the manual release workflow)
- `web/js/app/lnm-tasks.js` — LNM pipeline inventory and stage status helpers
- `web/js/app/atlas-options.js` — Selectable atlas registry. Maps the user-facing `Atlas` options to overlap atlas asset id, connectome asset id, weighting mode, affected-map label atlas id, colormap, and optional functional-profile asset.
- `web/js/app/lnm-labels.js` — Yeo 7-network, Schaefer400, and lesion-mask NiiVue colormaps
- `web/js/modules/overlap-export.js` — Pure JS CSV serialization for LNM overlap summaries
- `web/js/modules/overlap-render.js` — DOM rendering for the LNM network-overlap and affected-network results tables
- `web/js/modules/function-profiles.js` — Browser-side loader, weighted term ranking, and DOM rendering for exploratory Neurosynth/NiMARE functional association profiles
- `web/js/modules/ui/ConsoleOutput.js` — Timestamped log viewer helper used by both the default clinician-facing analysis log and the collapsed technical support log
- `web/js/app/sct-tasks.js` — SCT stable task inventory and task status helpers
- `web/js/app/labels.js` — Task labels + NiiVue colormap
- `web/js/modules/parcel-overlap.js` — Pure JS parcel and network overlap reducers
- `web/js/modules/threshold.js` — Threshold + cluster-cleanup helper for scalar maps (pure helper still supports absolute / percentile modes, but the connectivity-map UI uses top-percent only), optional symmetric/magnitude ranking, optional minClusterVoxels via existing CC helpers
- `web/js/modules/mask-transform.js` — Pure JS binary lesion-mask resampling/writing helpers used to project editable native drawings onto the fixed `lnm-mni160` grid while preserving the fixed output header
- `web/js/modules/spatial-file.js` — Lightweight spatial metadata registry for File-like NIfTIs. Tags volumes with logical spaces such as `native-t1`, `mni160`, or `atlas:<assetId>` plus optional dims/affine, and provides assertion helpers for computation inputs and viewer stacks.
- `web/js/modules/brain-extraction.js` — SynthStrip wrapper for manifest asset `lnm-synthstrip` (`models/synthstrip.onnx`, `SynthStrip-Hoopes2022-Apache-2.0`): RAS→LIA, foreground crop, adaptive fast-mode target spacing, conform/pad, ONNX SDT inference, original-grid SDT thresholding, largest-CC cleanup, and LIA→RAS mask output
- `web/js/modules/resample.js` — Affine-aware 3D resampler that bridges the SynthMorph 160×160×192 1mm warp output onto the selected atlas grid; `affineFromHeader` reads sform/qform off a nifti-reader-js header, `resampleAffine` does NN/trilinear sampling under the destination affine
- `web/js/modules/prealign.js` — In-browser affine pre-registration to the SynthMorph-required MNI160 1mm grid. `centroidOfMask` computes the brain-mask voxel centroid; `applyAffineToVoxel` applies a 4×4 affine to a voxel coord; `computePrealignAffine` builds the centroid-only destination affine (Phase 16 v1); `covarianceOfMask` + `jacobiEigen3x3` + `principalAxisAlign` add Phase 26 PCA rotation. `principalAxisAlign` transforms mask covariance into source world millimetres, assigns PCA eigenvectors to canonical X/Y/Z by source-affine world direction rather than eigenvalue size, enforces det(R)=+1, and composes the source-world-from-MNI-world transform with the fixed MNI affine so submillimetre source voxel sizes do not leak into the 1mm output grid.
- `web/js/inference-worker.js` — Module Web Worker running SynthStrip, SynthStroke, benchmark-gated DeepISLES DWI/ADC seed inference, SynthMorph registration, warp, inverse-warp, and label-map projection stages.
- `web/js/controllers/MaskDrawingController.js` — Thin NiiVue drawing wrapper for editable lesion masks: blank/seed drawing setup, paint/erase/erase-cluster pen values, brush/shape/fill controls, undo, smoothing/interpolation, drawing export/download, and drawing-mode cleanup
- `web/js/controllers/` — FileIO, DICOM, Inference, Viewer controllers
- `web/js/modules/` — UI components and inference pipeline modules

## Key Conventions

- The inference worker is a **module worker** (`type: 'module'`); load nifti-reader-js lazily inside the message handler (`niftiReady` promise) — top-level `await import(...)` causes Chromium to drop the first messages.
- Visible project branding is app name `CALMaR` with tagline `Co-designed Automated Lesion Mapping and Reporting`. Keep existing internal `lnm-*` ids, protocol headers, and filenames stable for compatibility unless a task explicitly asks to migrate them.
- The start page is a first-run explanation layer, not a separate marketing site. It must keep a clear `Start analysis` path into the existing workspace, explain the three-step workflow, and state that patient-derived data stays local while public atlas/model assets may be fetched.
- The start page must mention that background execution is possible when the site is added under `"Always keep these sites active"` in browser settings.
- The start page should fit into one viewport without requiring scroll on the default desktop-sized browser; keep hero spacing, local-processing copy, and the three-step workflow compact rather than expanding it like a landing page.
- The start page and workspace headers must include a VesselBoost-style `More Apps` header link to `https://neurodesk.org/getting-started/hosted/webapps/` with title `More Neurodesk web apps`, the shared 2x2 grid icon, and the same `.header-link` styling as the other header actions.
- Cache Storage (Atlas + connectome assets) requires URL-shaped keys. Bare strings like `'yeo7-fc-pack-adhd200-n30-v1'` parse as URL schemes and the put fails. Fold the manifest's `cacheKey` into the URL fragment: `${url}#${encodeURIComponent(cacheKey)}`. See `atlas-loader.js fetchCacheFirst`.
- The visible atlas control is a select field labelled exactly `Atlas`. Keep both `Schaefer 400 parcels` and `Yeo 7 networks` available. Schaefer400 is the default now that direct overlap, the 4 mm companion label atlas, and the lazy development-fMRI FC shards are supported; Yeo remains selectable for compatibility and comparison.
- Schaefer parcel display labels must omit the leading `7Networks_` prefix everywhere user-facing, including direct-overlap tables, affected-map tables, CSV export, and functional-profile driver labels. Keep numeric label ids and 7-network membership metadata intact for weighting/grouping.
- `ViewerController` owns NiiVue overlay lifecycle. `loadVolumeStack()` loads the base via `nv.loadVolumes([single])` and adds each overlay via `nv.addVolumeFromUrl()` — passing multiple entries to `loadVolumes()` is a silent regression in NiiVue 0.68.x (cal_min/cal_max/colormap LUT not initialised on the overlay path).
- Every app-created or app-loaded NIfTI that crosses a space boundary should be tagged through `tagSpatialFile()` / `tagDecodedFileSpace()`. Use `assertSpace()` before computation stages with fixed expectations (registration source, atlas overlap, threshold labeling) and `assertSameSpace()` / `assertVolumeStackSpaces()` before rendering overlays. Viewer checks must compare overlays against the active viewer base (`viewerBaseFile`), not just `structuralFile`, because mask-review mode displays native T1 while the pipeline may retain the prealigned MNI160 T1.
- SynthMorph deformable still expects the source T1 prealigned to the canonical 160×160×192 1mm `lnm-mni160` grid, and `stepRegister` enforces that input. `prealignToMni160()` may use an oblique PCA sampling affine internally, but the emitted `lnm-prealign-t1.nii` and brain mask must be written with the fixed `lnm-mni160` affine/header so NiiVue and later affine resampling see a canonical target grid. The hosted browser ONNX graph is currently spatially retargeted to 48×64×80 (`lnm-synthmorph-mni-48x64x80.onnx`) so the first Conv3D activation stays under the browser budget; the worker downsamples source/reference to 48×64×80 before ONNX and upsamples the integrated 24×32×40 SVF-derived displacement back to 160×160×192.
- Brain extraction uses exactly one supported SynthStrip asset: `web/models/manifest.json` id `lnm-synthstrip`, runtime filename `models/synthstrip.onnx`, cache key `lnm-synthstrip-v1`, source version `SynthStrip-Hoopes2022-Apache-2.0`, hosted at `https://huggingface.co/datasets/sbollmann/lnm-webapp-models/resolve/main/models/synthstrip.onnx`, sha256 `7b8eeecf3793a6c4510b9f5270ecc03d9c3262d26e08d568203a651ab4b84074`. The ONNX asset was converted from FreeSurfer's official main SynthStrip model v1 checkpoint `synthstrip.1.pt` at `https://surfer.nmr.mgh.harvard.edu/pub/dist/freesurfer/synthstrip/models/synthstrip.1.pt`, sha256 `37417f802196186441aae3e7f385d94f8a98c64a88acaeaa2723af995c653e33`; it is not the no-CSF or pediatric checkpoint. Keep README, manifest upstream provenance fields, tests, and the in-app citations modal aligned whenever the asset changes.
- SynthStrip fast mode must not blindly downsample 1mm clinical T1s to 2mm. `chooseFastTargetSpacing()` uses the foreground bounding box to stay in the smallest safe conform bucket (typically ~192³) while preserving as much native detail as possible, and resampled runs threshold the linearly upsampled SDT on the original grid rather than nearest-upsampling a binary 2mm mask.
- The old full-resolution SynthMorph ONNX graph OOMed browser runtimes because its first activation was ~4.7 GiB. Keep `npm run test:synthmorph-browser-model` green whenever the registration manifest or converter changes; it pins the browser graph dimensions, cache key, provider routing, and activation budget. The current 48×64×80 graph still contains 3D MaxPool nodes, so `browserRuntime.executionProviders` is `['wasm']`; ORT WebGPU rejects NHWC 3D pooling.
- Lesion segmentation uses exactly one default SynthStroke asset: `web/models/manifest.json` id `lnm-stroke-lesion`, runtime filename `models/lnm-stroke-lesion.onnx`, cache key `lnm-stroke-lesion-synthstroke-baseline-v1`, source version `SynthStroke-baseline-MELBA2025-MIT`, hosted at `https://huggingface.co/datasets/sbollmann/lnm-webapp-models/resolve/main/models/lnm-stroke-lesion.onnx`, sha256 `49f3d047a2299501791c7176cfda770d45d2f411a18ef41dd1105be4d0582cb8`. The ONNX asset was converted from `liamchalcroft/synthstroke-baseline` `config.json` + `model.safetensors`, not SynthStroke SynthPlus; observed upstream revision `b693a650026359705688fbce409219c4dbb5d6be`, config sha256 `2d9e7eb2ab4cb0a696ce6a845ad3123b77b31867f7ba74a271b81132cca38b1e`, weights size `74468100`, weights ETag `d56c089e8c4bcc0ad2281f1e80b7c0e265f3b7138dee17fb2d160487604eee66`. `scripts/convert_lesion_seg_model.py` reconstructs the MONAI UNet, loads Safetensors weights, traces static `1x1x128x128x128`, exports opset 17, and checks PyTorch-vs-ORT max abs diff `<1e-2`. Keep README, manifest upstream provenance fields, tests, and the in-app citations modal aligned whenever this asset changes.
- 2-channel softmax outputs (e.g. SynthStroke baseline and DeepISLES NVAUTO) must be collapsed to a single stroke probability/logit with the correct stroke channel before thresholding. Treating them as a 1-channel logit produces whole-brain coverage as a regression.
- Lesion segmentation runs with 8-axis test-time augmentation enabled (`testTimeAugmentation: true`) at sliding-window `overlap: 0.25`; both are read from the `lnm-stroke-lesion` manifest entry (`entry.testTimeAugmentation ?? true`, `entry.overlap ?? 0.25`) by `runLesionSegmentation()`. TTA averages reflection-flipped predictions to suppress SynthStroke false positives that are not stable under reflection — notably the in-brain cerebellum/posterior-fossa cluster reported on real T1s. On the prealigned MNI160 input, TTA at overlap 0.25 removes the cerebellar FP while preserving ~95% of the true lesion; raising overlap to 0.5 only costs recall, and the upstream 192³ config is both worse here and not browser-shippable (the ONNX is a static-128³ export). Do NOT "fix" lesion FPs by feeding a skull-stripped/brain-masked T1 to this model: it is hypersensitive to whole-volume z-score normalization, so zeroing non-brain voxels destabilises it (on the native grid it explodes to a whole-FOV over-segmentation; even on MNI160 it discards ~85% of the true lesion). TTA costs ~8× patch inference but is bounded and uses the existing ONNX.
- DeepISLES is currently an opt-in, benchmark-gated DWI/ADC seed path, not a default model swap. The app may collect DWI/TRACE and ADC inputs and expose a `DeepISLES DWI/ADC seed` action, but the manifest asset `lnm-deepisles-nvauto-browser-seed` must stay `benchmark-only` until the Dice 0.5 gap analysis identifies a browser-feasible candidate that passes SOOP metrics and browser budget checks. Do not silently feed a DWI-space seed into T1-based CALMaR mapping unless the DWI and native T1 grids are explicitly compatible.
- F-order NIfTI vs row-major NDHWC tensor layout: registration.js operates on F-order voxel arrays; SynthMorph forward consumes NDHWC; the worker handles both transpositions. Do not change the layout contract without updating `test_registration.cjs` + the parity fixture.
- FC packs are stored as NumPy row-major channels. `decodeFcPack()` transposes each map to NIfTI x-fast order; Yeo loads the whole 7-channel pack, while Schaefer uses `loadConnectomeChannelsFromManifest()` to fetch only shards for parcel labels hit by the lesion. `runFcNetworkMap()` writes network-map and threshold-map NIfTIs with the selected FC atlas affine; otherwise NiiVue renders striped or spatially misplaced scalar overlays.
- Bridge from MNI160 1mm warp output to the selected atlas grid: `applyRegistrationToLesion()` invokes `runWarpMask` (worker), awaits the `'mni-lesion'` stage data via a one-shot resolver, and resamples onto `getAtlasOption().overlapAtlasAssetId` with `resampleAffine(..., 'nearest')`. The worker must wrap `'mni-lesion'` with the fixed `lnm-mni160` reference header because those voxels are on the target grid, not the source/prealign header. The output replaces `this.lesionFile` so downstream stages run unchanged.
- Atlas-grid lesion files produced by `applyRegistrationToLesion()` are computation inputs for overlap/FC, not patient-space overlays. Do not route them through `setLesion()` when a structural T1 is loaded; that renders atlas-space voxels on the patient viewer stack and makes the mask appear outside the brain. Keep the confirmed `lesionMaskFile` / native lesion overlay for patient-space viewing.
- Manual lesion-mask refinement: automatic SynthStroke segmentation is only an editable seed. Advanced controls must offer an explicit choice between `Auto seed mask`, blank `Manual mask`, and `Upload mask`; the uploaded path treats the chosen NIfTI as a native-T1 editable seed, displays it on the anatomical scan, and routes it through the same drawing toolbar before `Confirm mask`. While `maskReviewActive=true`, the viewer must show a persistent `Mask approval required` banner as a visual prompt only; keep `Upload`, `Confirm mask`, and `Download` actions in the mask drawing toolbar above, with `Upload` and `Download` grouped as file actions, not duplicated inside the banner. The upload picker must accept only `.nii` / `.nii.gz`, and failed uploads should name the rejected file plus the expected NIfTI formats. Mask-review download and confirmation must synchronously write a native-space NIfTI from the active NiiVue `drawBitmap`; do not rely on async NiiVue `saveImage` downloads, which can lose browser click activation or produce empty files. Do not use `showSaveFilePicker()` / `createWritable()` for mask downloads in the in-app browser; it can open a save dialog but then fail writes, leaving zero-byte files. Instead, post the raw NIfTI bytes to the localhost `web/run.sh` `__lnm_downloads` route with `X-LNM-Stage-Only: 1`, then trigger a normal same-origin HTTP attachment download so the browser handles the save location. Static-hosted fallback paths may use the service-worker `Cache Storage` route or a data URL. Avoid blob URLs for mask downloads because the in-app browser can save them as empty files. `runFullPipeline()` pauses after the seed is loaded into NiiVue drawing mode on the native T1; downstream registration/warp/overlap/FC/threshold stages must not run until `Confirm mask` writes the drawing through the native NIfTI writer, binarizes it, resamples it through the saved `prealignSamplingAffine`, writes `lesionMaskFile` with the fixed `lnm-mni160` header, sets `lesionMaskConfirmed=true`, clears the editable NiiVue drawing overlay, loads the confirmed native lesion mask as a normal lesion overlay on the anatomical scan, and hides the mask-review toolbar.
- Lesion-mask review displays the native T1 as the viewer base. The Brain mask layer must start off in this mode. If the user manually enables it during review, use `nativeBrainmaskFile`; do not load the prealigned `lnm-prealign-brainmask.nii` on top of the native T1. When entering review, clear the old viewer stack so stale MNI-space brain-mask volumes cannot stay visible, hide NiiVue's 3D render tile, and suppress direct 3D view selection until review exits. Outside review mode, `brainmaskFile` remains the current pipeline-space mask.
- Mask drawing controls must use explicit destructive labels. The cluster-removal tool is labelled `Erase cluster` and describes that it erases the connected lesion cluster under the cursor; do not shorten it to ambiguous `Cluster`.
- Mask review toolbar controls are grouped by workflow: drawing mode/brush/shape/fill, edit actions (`Undo`, `Blank`, `Smooth`), slice interpolation (`Axial`/`Coronal`/`Sagittal` + `Interp` + compact help), then confirmation/file actions. `Interp` help must explain that it uses the selected orientation, finds the first and last non-empty drawn slices, runs NiiVue mask interpolation between them, and thresholds back to lesion voxels.
- `Blank` must preserve undo history when it clears an existing editable mask. It should clear the current same-sized `drawBitmap` in place and record undo snapshots, not close/recreate the NiiVue drawing and lose the previous mask state.
- `prealignToMni160()` must preserve both the native T1 geometry (`nativeStructuralFile` / native dims+affine) and the PCA/centroid sampling affine used for prealign. The source brain centroid is aligned to the fixed MNI template foreground centroid, not just the raw volume midpoint. That affine is the mapping contract for auto MNI160 seed → native drawing mask and edited native drawing mask → fixed MNI160 `lesionMaskFile`.
- While automatic lesion segmentation runs after prealign, the viewer should show the native T1, force the Brain mask layer off, and suppress NiiVue's 3D render tile. The worker still receives the prealigned MNI160 T1; this viewer-only state avoids exposing the transient prealign brain-mask QC overlay as if it were segmentation output.
- Registration should pass the prealigned brain mask into `run-register`. With `brainMaskBuffer`, the worker robustly normalizes source/target intensities inside foreground masks and zeros non-brain source voxels before SynthMorph; without a mask, the no-mask path preserves the previous self-pair behavior.
- Patient-space final threshold display: `applyNetworkThreshold()` creates the selected-atlas-grid threshold NIfTI, but if a structural T1 and SynthMorph displacement are available the app resamples that mask back to the fixed `lnm-mni160` registration grid (use the template affine, not the subject/prealign affine) and calls worker `inverse-warp-mask` to emit `threshold-patient`.
- Subject-space atlas QC: `showSubjectSpaceAtlas()` resamples the selected overlap atlas onto the fixed `lnm-mni160` grid, then calls worker `inverse-warp-mask` with `labelMap: true` so integer atlas labels survive the patient-space inverse warp. Schaefer labels require the Uint16 label-map path; do not collapse them to Uint8. The output stage is `atlas-patient`, rendered as viewer stage `atlas-qc` with the selected atlas colormap.
- Registration QC has four visual modes and must stay visual-only: patient space (subject T1 + inverse-warped full selected atlas), MNI space (fixed `lnm-mni160` template + registered T1 only), checkerboard (fixed template alternating with registered T1 only), and displacement (SynthMorph displacement magnitude). MNI and checkerboard QC must not add the atlas overlay; they are for comparing the subject brain to the atlas/template brain. `stepRegister()` emits `registered-t1-mni160` and `registration-displacement-mag` stage data for these views; the Advanced selector defaults to MNI space because that is the view where patient/template blending is meaningful.
- The Advanced `Patient/MNI blend` slider controls the `registered-t1-mni160` stage opacity over the fixed `registration-template` base so users can slide between MNI-only and registered-patient-only views when checking SynthMorph alignment. If registration artifacts exist but the active viewer is still patient-space, slider input switches the QC view to MNI space before applying the opacity.
- Advanced controls are an ordered manual workflow: brain extraction → pre-align T1 → choose `Auto seed mask`, blank `Manual mask`, or `Upload mask` → edit/confirm lesion mask → MNI registration → choose Registration QC view → check atlas alignment → adjust Patient/MNI blend → warp lesion to atlas grid → compute atlas overlap → compute network map. The `Check atlas alignment` button uses `showRegistrationQc()` and is a visual QC overlay only, not an automated pass/fail score.
- `runFullPipeline()` has two branches. (a) Manual: a selected-atlas-grid lesion mask is already loaded → skip seg/register/bridge. This branch is retained as a hidden/programmatic compatibility path, not a visible `Researcher mode` upload in Advanced controls. (b) Auto: structural T1 only → brain extraction, prealign, auto seed segmentation, then pause for manual lesion-mask review; `Confirm mask` resumes the remaining stages.
- The `outsideAtlasWarning` DOM id is retained for compatibility, but the visible UI copy is a neutral atlas-label coverage note. `voxelsOutsideAtlas` means lesion voxels where the selected atlas label is `0` (unlabeled by that atlas), not voxels outside a whole-brain mask. Do not imply either Yeo or Schaefer is a named language-area atlas.
- Worker-backed stages used by `runFullPipeline()` must resolve only after their emitted output is ready: brain extraction waits for `'brainmask'` stage data, auto seed segmentation waits for `'segmentation'` stage data, and registration waits for the `'register'` step-complete event before the warp bridge runs.
- Loading a structural image must never start SynthStrip or any other processing automatically. `setStructural()` may load the viewer and auto-select `lnm-yeo-auto`, but computation starts only from `Run analysis` or an explicit per-stage button.
- Do not expose a visible pipeline selector. `Run analysis` is input-driven: structural T1 selects the full auto chain, and researcher-mode atlas-grid masks select the hidden manual network-map path; advanced users can still run individual stages with the compact per-stage buttons.
- Keep sidebar help copy behind compact inline `i` help popovers (`.help-icon` / `.help-popover`) rather than always-visible helper paragraphs, matching the QSMbly-style controls.
- Logging has two viewer surfaces: the default `consoleOutput` analysis log is clinician-facing and should use a strict allowlist of condensed major workflow messages plus actionable warnings/failures; file-load mechanics, viewer mechanics, PCA geometry, model/runtime chatter, timing, shape, and processing details belong only in the collapsed `technicalConsoleOutput`, which must retain the original full messages for support/debugging.
- Results copy must preserve the distinction between `Direct lesion overlap` (selected atlas labels containing lesion voxels directly; `% of lesion`) and `Connectivity-map effects` (selected atlas labels containing voxels that survive thresholding of the group-FC weighted connectivity map; `% of map`).
- Functional profile copy must stay exploratory: Yeo7 term profiles come from the compact `yeo7-neurosynth-v7-function-profiles` annotation asset; Schaefer400 term profiles come from `schaefer400-neurosynth-v7-function-profiles`, generated by parcel-wise NiMARE `ROIAssociationDecoder` over all 400 Schaefer ROIs. These are combined with either direct lesion-overlap weights or thresholded connectivity-map weights and must be described as literature associations rather than clinical predictions. The `Score` column carries the numeric weighted term score; driver columns list only contributing network or atlas-label names so single-label rows do not repeat the same score twice.
- Keep advanced per-stage button labels compact. Technical grid details such as MNI160 1 mm belong in `title`/help text, not inline superscript/unit strings that wrap inside small buttons.
- Threshold UI: the connectivity-map surface is top-percent only; do not expose absolute t-stat threshold mode. The slider uses user-facing top-percent semantics (`5` keeps roughly the strongest 5%, `0` keeps none), and `applyNetworkThreshold()` always converts that to the percentile cutoff expected by `applyThresholdDetailed`. The optional magnitude checkbox ranks by `|t|`; without it, only strongest positive values are kept. The min-cluster field lives at the beginning of the Results section, defaults to 30 voxels, filters all following result tables/functional-profile drivers, and also re-fires thresholding on every input change. The summary reports how many voxels cluster cleanup removed, so "cluster size" changes that do not affect a large connected component are visible instead of silent. The thresholded mask, final affected-atlas table, summary, and live red `threshold-preview` overlay stay in sync while the scalar FC map remains visible.
- Viewer layer toggles are stage-driven and must hide volumes by setting opacity to `0`, not by removing NiiVue volumes. `ViewerController` preserves per-stage visibility for `structural`, `brainmask`, `segmentation`/`lesion`, `threshold-preview`, and `atlas-qc` so toggles survive overlay replacement and threshold-slider refreshes. Checkbox checked state must follow the currently loaded viewer stages, not stale app files; if a layer file exists but is not loaded in the active stack, leave its checkbox enabled but unchecked so the user can add it back explicitly. Available brain-mask, lesion-mask, or atlas overlays can be reactivated from unchecked controls. Lesion-mask volume overlays use the dedicated blue `lnm-lesion-blue` colormap; editable lesion drawings pass the same blue colormap object directly to NiiVue because `setDrawColormap()` does not resolve app-registered custom colormap names. Threshold-map previews remain red. During active mask review, the `Lesion mask` toggle also controls the editable NiiVue drawing opacity because the drawing is not a normal volume stage. Truly unavailable layer toggles are disabled and unchecked; for example, `Atlas` must not appear checked before an atlas-QC volume has been generated/loaded or can be generated from registration.
- Config version is bumped by the manual GitHub Actions release workflow via `sed`; it increments the patch version — do not bump manually (per-phase commits in this codebase have done so explicitly under TDD).
- `populateVersionLabel()` uses `formatVersionLabel()` to combine `Config.VERSION` with `build-info.json`; staging versions already carry `-staging+<shortsha>`, so the build-info SHA must not be appended a second time. Production deploys set `build-info.json` `branch` to the release tag (e.g. `v0.17.13`), which equals the version; `formatVersionLabel()` suppresses a `branch` that is the version tag (or the `detached`/`main` fallbacks) so the label reads `v0.17.13 (2b26f96)`, not `v0.17.13 (2b26f96, v0.17.13)`.

## CI/CD

- **Release workflow** (`.github/workflows/release.yml`): manual-only promotion. The `validate` job runs the full `npm test` (including heavy ONNX-inference tests), captures failed-test details in the Actions step summary/job outputs, and the `release` job only runs on green and bumps version, creates tag + GitHub release.
- **Deploy workflow** (`.github/workflows/deploy-pages.yml`): deploys staging from `main` immediately on pushes to `main`, and deploys production from the latest release tag after the manual release workflow completes successfully. It downloads ONNX Runtime WASM files and verifies model assets before deploying to GitHub Pages; tests are run by the release workflow, not the deploy workflow.
- GitHub Pages deploys must check out Git LFS assets and verify `web/models/*.onnx` are real model binaries, not LFS pointer files.
- Production deploys build from the latest release tag while the workflow file comes from `main`; asset verification must tolerate older release tags by validating ONNX files and template `.nii.gz` files that exist in the checked-out build, without hard-coding newer template paths.
